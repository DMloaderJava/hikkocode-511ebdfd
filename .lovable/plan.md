
## Новая архитектура агента

Вместо текущего подхода "отправить всё в один запрос и получить все файлы назад", внедряем **пошаговый цикл с инструментами (tool-calling)**:

### Фазы работы агента:

**1. Index (при создании/открытии проекта)**
- Отправляем структуру проекта (пути файлов, размеры, языки) в Gemini
- Получаем краткое описание каждого файла → сохраняем в `file_index`
- Это даёт агенту "карту проекта" без загрузки всех файлов в контекст

**2. Thinking (размышление)**
- Агент получает запрос пользователя + карту проекта (индекс)
- Gemini рассуждает: какие файлы нужно прочитать, что менять
- Вывод стримится в UI как "thinking" блок

**3. Read (чтение файлов)**
- Агент вызывает tool `read_file(path)` для конкретных файлов
- Содержимое добавляется в контекст
- В UI отображается: "Read src/pages/Landing.tsx"

**4. Understand (понимание картины)**
- После чтения — отдельный запрос на анализ
- Агент формирует план изменений с обоснованием
- В UI: список найденных проблем/задач

**5. Edit (редактирование)**
- Агент вызывает tool `edit_file(path, changes)` для каждого файла
- Изменения применяются инкрементально
- В UI: "Edited Landing.tsx" с диффом

### Реализация:

- **Edge function** `supabase/functions/agent-v2/index.ts` — новый оркестратор с tool-calling
- **Клиентский контроллер** `src/lib/agentControllerV2.ts` — обработка стрима с фазами
- **UI компонент** обновлённый `AgentTasksPanel` — отображение thinking/read/edit шагов

### Модель: `google/gemini-2.5-flash` с tool-calling

Tools доступные агенту:
- `list_files()` — список файлов проекта
- `read_file(path)` — прочитать содержимое файла
- `edit_file(path, content)` — заменить содержимое файла
- `create_file(path, content)` — создать новый файл
- `delete_file(path)` — удалить файл

### Что изменится в UI:
- Шаги агента отображаются как в Lovable: Thinking → Read file1 → Read file2 → Understanding → Edit file1 → Edit file2
- Каждый шаг кликабельный с деталями
